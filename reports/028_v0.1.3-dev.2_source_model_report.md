# Mission 028 - 0.1.3-dev.2 - Socle modele sources locales/distantes

## Resume

Mission preparatoire realisee sur la branche de travail `0.1.3-dev.1`, avec passage en `0.1.3-dev.2`.

Objectif : preparer le modele de donnees pour distinguer les images locales des futures images distantes, sans ajouter d'import URL, sans racine Web visible, sans Midjourney, sans recuperation reseau et sans changement utilisateur visible.

Constat principal :

La table `images` reste aujourd'hui structurellement locale :

- `root_id INTEGER NOT NULL`
- `path TEXT UNIQUE NOT NULL`
- `folder_path TEXT NOT NULL`
- `file_name TEXT NOT NULL`

Il ne faut donc pas creer d'image distante reelle tant que ces contraintes ne sont pas traitees dans une mission specifique. La mission 028 ajoute le socle de distinction et la table distante, mais ne force aucun faux chemin local.

## Choix de modele retenus

### `images.source_kind`

Ajout de la colonne :

```sql
source_kind TEXT NOT NULL DEFAULT 'local' CHECK(source_kind IN ('local', 'remote'))
```

Valeurs retenues :

- `local`
- `remote`

Choix volontaire :

- pas de `midjourney` dans `source_kind` ;
- Midjourney restera plus tard un fournisseur ou groupe distant specialise ;
- toutes les images actuelles et nouvellement scannees restent `local`.

### `remote_images`

Ajout d'une table dediee aux futures informations distantes :

```sql
CREATE TABLE IF NOT EXISTS remote_images (
  image_id INTEGER PRIMARY KEY,
  remote_url TEXT NOT NULL,
  provider TEXT,
  provider_id TEXT,
  provider_group_id TEXT,
  remote_slot TEXT,
  source_status TEXT NOT NULL DEFAULT 'remote' CHECK(source_status IN ('remote', 'cached', 'archived')),
  last_checked_at TEXT,
  last_known_status TEXT,
  created_at TEXT,
  updated_at TEXT,
  FOREIGN KEY(image_id) REFERENCES images(id)
);
```

Cette table isole les details distants sans polluer `images` avec des colonnes specialisees.

## Migration des images existantes

Pour les bases existantes, `migrateDatabaseSchema()` :

1. conserve la migration existante de `image_user_meta.workflow_color` ;
2. ajoute `images.source_kind` si la colonne n'existe pas ;
3. applique `DEFAULT 'local'` ;
4. normalise les valeurs vides ou nulles vers `local`.

Les nouvelles bases creent directement `source_kind` via `db/schema.sql`.

La table `remote_images` est creee par `CREATE TABLE IF NOT EXISTS` lors de l'execution du schema.

## Creation et mise a jour des images locales

La fonction `persistImagesForFolder` ecrit maintenant explicitement :

- `source_kind = 'local'` a l'insertion ;
- `source_kind = 'local'` lors d'un `ON CONFLICT(path) DO UPDATE`.

Ainsi, tout scan local continue de produire uniquement des images locales.

## Representation future d'une image distante

Etat apres mission 028 :

- `images.id` peut servir d'identite commune ;
- `remote_images.image_id` peut porter les informations distantes ;
- `source_kind` permet de distinguer `local` et `remote`.

Limite importante :

Une image distante ne peut pas encore etre inseree proprement dans `images` sans une migration supplementaire, car `root_id`, `path` et `folder_path` restent obligatoires et locaux.

Ce choix evite volontairement d'introduire :

- un faux chemin ;
- un faux dossier ;
- un faux fichier ;
- une racine locale artificielle.

La prochaine mission de modele devra donc decider comment representer une ligne `images` distante sans `path` local, probablement en rendant certains champs locaux optionnels ou en separant plus explicitement les attributs locaux.
