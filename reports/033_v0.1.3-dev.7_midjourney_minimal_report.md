# Mission 033 - 0.1.3-dev.7 - Midjourney minimal

## Resume

Mission realisee depuis `0.1.3-dev.6`, avec passage en `0.1.3-dev.7`.

Objectif atteint : ajouter une premiere integration Midjourney minimale permettant de coller une URL CDN Midjourney ou un job ID UUID, puis de creer les quatre images distantes du job dans `images` + `remote_images`.

## Commande UI ajoutee

- `Fichier > Ajouter un job Midjourney...`

Cette commande ouvre une modale simple avec un champ :

```txt
URL Midjourney ou job ID
```

## Validation Midjourney retenue

Validation cote main process uniquement.

Formats acceptes :

- UUID/job ID seul :

```txt
32f08e2c-8188-4a08-bd93-89d22369d3ad
```

- URL CDN Midjourney stricte :

```txt
https://cdn.midjourney.com/32f08e2c-8188-4a08-bd93-89d22369d3ad/0_1.png
```

Regles :

- protocole `https://` obligatoire ;
- domaine obligatoire `cdn.midjourney.com` ;
- job ID au format UUID ;
- slot limite a `0_0.png`, `0_1.png`, `0_2.png`, `0_3.png` ;
- refus des chaines vides, domaines tiers, slots invalides et schemes non HTTPS.

## Extraction du job ID

- Si l'entree est un UUID seul, il est utilise directement comme `jobId`.
- Si l'entree est une URL CDN, le job ID est extrait du premier segment du chemin :

```txt
https://cdn.midjourney.com/<job_id>/<slot>.png
```

Le slot detecte dans l'URL est valide, mais la mission cree toujours les quatre slots du job.

## Generation des 4 URLs

Pour un job ID valide, Iconotheque genere :

```txt
https://cdn.midjourney.com/<job_id>/0_0.png
https://cdn.midjourney.com/<job_id>/0_1.png
https://cdn.midjourney.com/<job_id>/0_2.png
https://cdn.midjourney.com/<job_id>/0_3.png
```

## Creation dans SQLite

Pour chaque slot :

- creation ou reutilisation d'une ligne `images` avec `source_kind = 'remote'` ;
- creation ou mise a jour de la ligne `remote_images`.

Valeurs Midjourney :

- `provider = 'midjourney'`
- `provider_id = <job_id>`
- `provider_group_id = <job_id>`
- `remote_slot = '0_0' | '0_1' | '0_2' | '0_3'`
- `source_status = 'remote'`

Aucune ligne `local_images` n'est creee par ce parcours.

## Deduplication

La deduplication repose sur `remote_images.remote_url`.

Comportement :

- si l'URL exacte existe deja, elle n'est pas recreee ;
- si le job existe partiellement, seuls les slots manquants sont crees ;
- si une URL Midjourney avait ete ajoutee comme image web generique, elle est qualifiee en `provider = 'midjourney'` pour apparaitre dans la vue Midjourney.

Le resultat IPC retourne :

- `jobId`
- `createdCount`
- `existingCount`
- `imageIds`
- `images`

## Vue Web / Midjourney

La colonne gauche contient maintenant deux entrees Web :

```txt
Web / Images par URL
Web / Midjourney
```

`Web / Images par URL` liste les images `provider = 'generic_url'`.

`Web / Midjourney` liste les images `provider = 'midjourney'`.

Les images Midjourney restent affichees comme images individuelles, sans vue groupee par job.

## Badges et informations

La grille affiche un badge compact :

- `MJ 0_0`
- `MJ 0_1`
- `MJ 0_2`
- `MJ 0_3`

Le panneau d'informations affiche :

- source `Midjourney` ;
- slot ;
- groupe/job ID ;
- URL applicative `iconotheque-image://remote/<imageId>`.

L'URL distante brute n'est pas utilisee comme source principale dans le renderer.

## Metadonnees

Les metadonnees utilisateur continuent d'utiliser :

```txt
image_user_meta.image_id -> images.id
```

Chaque slot Midjourney peut donc recevoir ses propres metadonnees : note, favori, description, tags/termes, couleur de workflow et autres champs deja existants.
