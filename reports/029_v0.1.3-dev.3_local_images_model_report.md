## Schema final

### `images`

`images` reste le catalogue commun par son identite :

- `id`
- `source_kind`
- anciennes colonnes locales heritees conservees temporairement

Important :

Les anciennes colonnes locales de `images` restent physiquement presentes pour compatibilite de migration. Le code nouveau ne doit plus les considerer comme la source conceptuelle des details locaux.

### `local_images`

Nouvelle table :

```sql
CREATE TABLE IF NOT EXISTS local_images (
  image_id INTEGER PRIMARY KEY,
  root_id INTEGER NOT NULL,
  folder_id INTEGER,
  path TEXT NOT NULL UNIQUE,
  folder_path TEXT NOT NULL,
  file_name TEXT NOT NULL,
  extension TEXT,
  size_bytes INTEGER,
  modified_at TEXT,
  preview_id TEXT,
  created_at TEXT,
  updated_at TEXT,
  FOREIGN KEY(image_id) REFERENCES images(id),
  FOREIGN KEY(root_id) REFERENCES roots(id),
  FOREIGN KEY(folder_id) REFERENCES folders(id)
);
```

`local_images.path` devient la reference locale applicative pour retrouver une image locale par chemin.

### `remote_images`

La table creee en mission 028 reste disponible :

```sql
remote_images (
  image_id INTEGER PRIMARY KEY,
  remote_url TEXT NOT NULL,
  provider TEXT,
  provider_id TEXT,
  provider_group_id TEXT,
  remote_slot TEXT,
  source_status TEXT NOT NULL DEFAULT 'remote',
  last_checked_at TEXT,
  last_known_status TEXT,
  created_at TEXT,
  updated_at TEXT
)
```

Elle n'est pas encore utilisee par une fonctionnalite visible.

## Choix sur les anciennes colonnes locales de `images`

Les anciennes colonnes restent temporairement dans `images`.

Raison :

- SQLite ne permet pas une suppression/reconstruction de colonnes complexes sans migration de table complete ;
- les contraintes existantes de `images` sont encore utilisees comme compatibilite pour les bases existantes ;
- supprimer ou reconstruire `images` dans cette mission augmenterait fortement le risque pour les metadonnees utilisateur deja liees a `images.id`.

Ce choix est volontairement transitoire.

Direction retenue :

- `images.id` reste l'identite commune ;
- `local_images` devient la source de verite applicative pour les attributs locaux ;
- une future mission pourra reconstruire ou alleger `images` lorsque la compatibilite aura ete stabilisee.

## Migration des donnees existantes

La migration `schema_version = 5` :

1. conserve les migrations precedentes ;
2. cree `local_images` si necessaire ;
3. normalise `images.source_kind` vers `local` si la valeur est vide ;
4. recopie les lignes locales existantes depuis `images` vers `local_images` ;
5. preserve les `images.id` existants ;
6. preserve les metadonnees, tags et termes lies a `images.id`.

La migration ne supprime aucune donnee.

La migration ne modifie aucun fichier image original.

## Fonctionnement de `persistImagesForFolder`

Apres mission 029, un scan local :

1. insere ou met a jour la ligne catalogue `images` avec `source_kind = 'local'` ;
2. recupere l'identite stable `images.id` ;
3. insere ou met a jour `local_images` avec les details locaux ;
4. conserve la deduplication par chemin local ;
5. ne cree aucune image distante.

Une compatibilite d'ecriture vers les anciennes colonnes locales de `images` est conservee tant que ces colonnes existent.
