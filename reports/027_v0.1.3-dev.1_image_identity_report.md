# Mission 027 - Ouverture 0.1.3-dev.1 et identite image stable

## Resume de la mission

Mission preparatoire pour ouvrir la version de travail `0.1.3-dev.1` et ajouter une identite stable aux images exposees au renderer.

Aucune source web n'a ete ajoutee. Aucune racine `Web`, aucun import URL, aucune integration Midjourney et aucune modification visuelle volontaire n'ont ete implementes.

## Version precedente

- `0.1.2`

## Nouvelle version

- `0.1.3-dev.1`

## Choix retenu

Le choix retenu est `imageId: number`.

Raison :

- la table SQLite `images` possede deja une cle primaire stable `id` ;
- les metadonnees utilisateur sont deja liees a `image_user_meta.image_id` ;
- ce choix prepare mieux les futures images non locales qu'une identite fondee uniquement sur le chemin disque ;
- `path` reste conserve dans `ImageFile` pour compatibilite, affichage, selection et comportement actuel.

Un type complementaire `ImageIdentity` a ete ajoute cote renderer :

```ts
{
  imageId: number;
  imagePath: string;
}
```

Le `imagePath` reste transmis comme fallback pour ne pas casser les parcours existants.

## IPC adaptes

Les retours `ImageFile` incluent maintenant `imageId`.

Parcours adaptes :

- `root-folder:list-images`
- `search:advanced`

Canaux de metadonnees prepares pour accepter une identite stable :

- `image-metadata:get`
- `image-metadata:save`
- `image-metadata:batch-update`

Le main process resout maintenant l'image via :

1. `imageId` si disponible et autorise ;
2. fallback `imagePath` si necessaire ;
3. rejet si l'image n'est pas autorisee pour la session.

## Points de compatibilite conserves

- `path` n'a pas ete supprime.
- Les API de metadonnees conservent un fallback par chemin.
- Le schema SQLite n'a pas ete modifie.
- Les images locales restent les seules sources affichees.
- Le protocole image local existant reste inchange.
- Les fichiers originaux ne sont ni modifies, ni renommes, ni deplaces, ni supprimes.

## Risques restants

- Plusieurs composants utilisent encore `path` comme cle de selection et de navigation ; c'est volontaire pour ne pas changer le comportement, mais une future integration web devra introduire une cle UI plus neutre si des images sans chemin local apparaissent.
- L'edition par lot accepte maintenant des identites enrichies, mais l'interface continue de calculer les selections avec les chemins.
- La recherche avancee retourne `imageId`, mais reste limitee aux images locales autorisees dans la session.
- Les futures sources web devront encore definir le modele `source_kind` / tables distantes avant usage.

## Recommandations pour la suite

- Avant d'ajouter l'import URL, introduire le modele de donnees distant de facon explicite : `source_kind` et table dediee `remote_images`.
- Garder `images.id` comme identite commune pour les metadonnees.
- Ajouter plus tard une cle UI neutre si les selections doivent melanger local et distant.
- Ne pas exposer d'URL distante brute au renderer sans validation et mediation par le main process.
- Faire une recette manuelle avec dossier local avant la premiere mission web pour confirmer que le comportement local est strictement inchange.
