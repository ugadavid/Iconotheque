## Migration ou reconstruction

La base de test a ete migree par reconstruction controlee.

Strategie appliquee :

1. creation / verification de `local_images` ;
2. copie des details locaux depuis l'ancienne table `images` vers `local_images` ;
3. creation de `images_new` avec le schema neutre ;
4. copie des `images.id`, `source_kind`, `created_at`, `updated_at` ;
5. remplacement de l'ancienne table `images` par la nouvelle table neutre ;
6. conservation des tables de metadonnees rattachees a `images.id`.

Aucun reset de base n'a ete necessaire.

Aucun fichier image original n'a ete modifie, deplace, renomme ou supprime.

## Schema final de images

```sql
CREATE TABLE IF NOT EXISTS images (
  id INTEGER PRIMARY KEY,
  source_kind TEXT NOT NULL DEFAULT 'local' CHECK(source_kind IN ('local', 'remote')),
  created_at TEXT,
  updated_at TEXT
);
```

`images` ne contient plus :

- `root_id`
- `folder_id`
- `path`
- `folder_path`
- `file_name`
- `extension`
- `size_bytes`
- `modified_at`
- `preview_id`

## Schema final de local_images

```sql
CREATE TABLE IF NOT EXISTS local_images (
  image_id INTEGER PRIMARY KEY,
  root_id INTEGER NOT NULL,
  folder_id INTEGER,
  path TEXT NOT NULL UNIQUE,
  folder_path TEXT NOT NULL,
  file_name TEXT NOT NULL,
  extension TEXT,
  size_bytes INTEGER,
  modified_at TEXT,
  preview_id TEXT,
  created_at TEXT,
  updated_at TEXT,
  FOREIGN KEY(image_id) REFERENCES images(id),
  FOREIGN KEY(root_id) REFERENCES roots(id),
  FOREIGN KEY(folder_id) REFERENCES folders(id)
);
```

`local_images.path` est maintenant la reference locale applicative pour le chemin fichier.

## Schema final de remote_images

```sql
CREATE TABLE IF NOT EXISTS remote_images (
  image_id INTEGER PRIMARY KEY,
  remote_url TEXT NOT NULL,
  provider TEXT,
  provider_id TEXT,
  provider_group_id TEXT,
  remote_slot TEXT,
  source_status TEXT NOT NULL DEFAULT 'remote' CHECK(source_status IN ('remote', 'cached', 'archived')),
  last_checked_at TEXT,
  last_known_status TEXT,
  created_at TEXT,
  updated_at TEXT,
  FOREIGN KEY(image_id) REFERENCES images(id)
);
```

`remote_images` reste pret pour les futures images distantes, mais n'est pas encore utilise fonctionnellement.

## Fonctionnement de persistImagesForFolder

Le scan local fonctionne maintenant ainsi :

1. normalise le chemin local ;
2. cherche une image existante via `local_images.path` ;
3. si elle existe, reutilise `local_images.image_id` ;
4. met a jour `images.updated_at` ;
5. si elle n'existe pas, cree une ligne neutre dans `images` avec `source_kind = 'local'` ;
6. recupere le nouvel `images.id` ;
7. insere ou met a jour `local_images` avec les details locaux.

Le dedoublonnage local se fait donc par :

```txt
local_images.path
```

et non plus par `images.path`.
