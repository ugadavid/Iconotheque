# Mission 034 - 0.1.3-dev.8 - Correction affichage images distantes

## Resume

Mission realisee depuis `0.1.3-dev.7`, avec passage en `0.1.3-dev.8`.

Objectif : corriger le rendu des images distantes, en particulier Midjourney, et eviter les vignettes silencieusement vides lorsqu'une image distante ne peut pas etre chargee.

## Cause identifiee

Les donnees Midjourney creees en mission 033 etaient correctes :

```json
[
  [1784,"remote","midjourney","https://cdn.midjourney.com/937064e2-8eec-4abe-858e-454a7f334f3f/0_3.png","0_3",0],
  [1783,"remote","midjourney","https://cdn.midjourney.com/937064e2-8eec-4abe-858e-454a7f334f3f/0_2.png","0_2",0],
  [1782,"remote","midjourney","https://cdn.midjourney.com/937064e2-8eec-4abe-858e-454a7f334f3f/0_0.png","0_0",0],
  [1781,"remote","midjourney","https://cdn.midjourney.com/937064e2-8eec-4abe-858e-454a7f334f3f/0_1.png","0_1",0]
]
```

Interpretation :

- `images.source_kind = 'remote'` : OK ;
- `remote_images.provider = 'midjourney'` : OK ;
- `remote_images.remote_url` : OK ;
- `remote_images.remote_slot` : OK ;
- `local_count = 0` : aucune ligne `local_images`.

Le probleme ne venait donc pas de la creation du job ni de la liste renderer.

Deux causes ont ete traitees :

1. le chargement distant utilisait le `fetch` Node/global depuis le main process, moins adapte a certains CDN que la pile reseau Chromium/Electron ;
2. le renderer masquait l'image en erreur avec `display: none`, ce qui laissait une tuile vide sans explication.

## Verification reseau

Test avec image generique :

```txt
https://httpbin.org/image/png
```

Resultat :

```txt
200 image/png 8090
```

Test avec une URL Midjourney deja stockee :

```txt
https://cdn.midjourney.com/937064e2-8eec-4abe-858e-454a7f334f3f/0_3.png
```

Resultat du fetch Node direct :

```txt
403 text/html; charset=UTF-8 5733
```

Cela confirme que l'URL stockee est bien appelee, mais que le chargement distant peut echouer selon le comportement du CDN si l'appel ne ressemble pas assez a un chargement navigateur.

## Correction appliquee cote protocole

Dans `electron/main.ts` :

- import de `net` depuis Electron ;
- remplacement du fetch distant par `net.fetch(...)`, qui passe par la pile reseau Electron/Chromium ;
- ajout de headers explicites :
  - `Accept: image/...`
  - `User-Agent` de type navigateur ;
- conservation du protocole controle :

```txt
iconotheque-image://remote/<imageId>
```

- ajout de `Access-Control-Allow-Origin: *` dans la reponse protocole ;
- retour d'un `Buffer` image avec `Content-Type` et `Cache-Control: no-store`.

Le renderer ne recoit toujours pas l'URL distante brute comme source principale.

## Logs ajoutes

Logs main process sobres :

```txt
[Iconotheque] remote image request imageId=...
[Iconotheque] remote image url=...
[Iconotheque] remote image response imageId=... status=... content-type=...
```

Ces logs permettent de distinguer :

- image absente en base ;
- URL refusee par validation ;
- HTTP non OK ;
- content-type non image ;
- erreur reseau/fetch.

## Comportement en cas d'image indisponible

Avant :

- l'image en erreur etait masquee ;
- la tuile restait vide.

Apres :

- la grille affiche :

```txt
Image distante indisponible
```

```txt
Image distante indisponible
```

Pour une image locale en erreur, le message reste generique :

```txt
Image indisponible
```

## Recommandation pour la suite
